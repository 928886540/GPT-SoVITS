# Distribution Plan

## 定位

本项目的交付目标不是公网服务，也不是作者长期托管 API。

目标是把可用产品一次性打包给社区用户，让用户在自己机器上运行：

- 本地模型
- 本地推理引擎
- 本地启动脚本
- 本地配置文件
- 可选 Tavo.js / 前端集成资源
- 示例音色和说明文档

## 交付原则

- 默认本机运行，不依赖远程服务器。
- 尽量少写 C 盘缓存。
- 路径尽量可迁移，不把作者机器的绝对路径写死。
- 用户启动过程尽量少命令，优先一键脚本。
- 能离线运行的部分尽量离线。
- 模型、参考音频、角色配置要有清晰目录结构。
- 大文件与源码/脚本分层，避免更新脚本时反复重发大模型。

## 可能交付形态

### 方案 A：完整整合包

包含：

- Python runtime 或明确依赖说明
- GPT-SoVITS/Genie 运行代码
- 模型文件
- 角色配置
- 启动脚本
- Tavo.js 或其他前端资源
- 示例和说明

优点：用户最省事。

缺点：包体大，更新成本高。

### 方案 B：轻量代码包 + 独立模型包

包含：

- 代码和脚本小包
- 模型/音频大包单独下载或单独分发

优点：后续更新轻，方便分渠道发。

缺点：用户需要放对目录。

### 方案 C：Tavo.js + 本地引擎说明

包含：

- Tavo.js
- 本地引擎启动说明
- 配置模板

优点：最轻。

缺点：对用户要求高，不适合小白。

## 当前优先验证

优先验证 `方案 B`：

- 代码和启动脚本保持小。
- 模型、音频、角色包单独管理。
- 后续可以再做完整整合包。

## 成功标准

- 用户能按说明在本机启动。
- 不需要作者提供远程服务。
- 不需要用户手动处理复杂依赖冲突。
- 多音色配置可复制。
- 中文和日语样例可直接跑。
- 流式或准流式播放能稳定工作。
- C 盘缓存增长可控。
